De volta ao básico - Por que HTML puro é o bastante

Uma ode à simplicidade e às raízes da internet

Henrique Gogó 2022-12-15

Possivelmente as coisas mais postadas em blogs desde que os inventaram são as tradicionais postagens "estou de volta" ou "agora vou escrever com mais rotina".

Este possivelmente é mais um destes. Mas deixe-me explicar antes de se tornar só mais um "blog de único post

Sempre fui uma pessoa minimalista. Desde os meus tempos mais remotos com o IBM 486SX que tínhamos em casa, eu buscava aplicativos e joguinhos que coubessem em apenas num disquete. "Se passar de 1.44MB, é grande e não vale a pena".

Nessa minha jornada pela internet, já iniciei (e abandonei) diversos outros blogs. Meu primeiro provavelmente foi no Blogspot. Com a popularidade do wordpress.com, não havia quem não tivesse um, e por muito tempo foi a plataforma principal de criação e publicação de blogs

Nesse tempo fui muito produtivo em posts. Foquei em posts de tecnologia e nessa época estava bem envolvido com a comunidade local de desenvolvimento. Reviews de eventos de tecnologia locais, versões novas de softwares, dicas de linux, tudo o que "vinha na telha".

Mas o verme do "early adopter" não larga a gente e assim que surgiu o Tumblr, não havia quem não usasse. Era como não ter uma conta no Instagram.

No Tumblr continuei minhas postagens técnicas, mas já com menos gás do que fazia no Wordpress. Dessa vez integrei uma forma de contato direto comigo pelo site e numa postagem que fiz sobre CouchDB, fui chamado para submeter um paper para o The Developers Conference. Funcionou! Meu blog estava finalmente "me deixando famoso".

Mas como tudo na vida, produzir exige tempo, e esse foi um tempo e pouco depois veio casamento. Um ano depois minha primeira filha. Depois contas. E "tempo livre" passou a ser destinado para outras coisas.

Mas o verme da curiosidade...

Geradores estáticos de páginas

Jerkyll, Hugo, Next.js, Jamstack e tantas sopas de letrinhas. Mas eu sou um cara minimalista, lembra? O que não significa que não seja um cara que goste de reinventar a roda.

Nos meus tempos de "programar HTML" offline, lembro que muito pouca tag era necessária para criar uma página realmente válida. Essencialmente muito menos do que pensamos. Sequer a tag html é necessária, embora o atributo lang seja altamente recomendado.

Abaixo o mínimo necessário para se ter uma página html válida: 

<!DOCTYPE html>
<html lang="en">
<title>Title</title>

Isso mesmo, sem head body. Na verdade o navegador irá se encarregar de fazer o "trabalho sujo". E sim, este é o mínimo que gera um código válido para os critérios da W3C.

Mas o mínimo não significa o melhor. Como sei que irei escrever em português e nossa língua possui acentos e caracteres que podem dar problemas de encoder dependendo do navegador dos visitantes, faz-se necessário adicionar uma declaração de charset.

Outra questão que nos dias de hoje se faz necessário é criar um viewport compatível com celulares. Não há nada mais chato do que abrir aquele site com letras minúsculas no celular.

Outra coisa que é óbvia: um site sem conteúdo não é propriamente um site com utilidade. A questão aqui é que não necessariamente precisamos declarar um body e toda a sua estrutura de aninhamento. Basta sair escrevendo abaixo do título utilizando tags simples de html. 

<!DOCTYPE html>
<html lang="en">
<meta charset="utf-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<title>Title</title>

<h1>My post title</h1>
<p>This is a post paragraph
<p>And this is another one

- Ei! Como você pode deixar tags abertas? Como pode ser tão descuidado?

- Calma, pequeno gafanhoto.

O interessante de se perceber aqui é que o navegador é inteligente o suficiente e é capaz de fechar inclusive tags obedecendo certa ordem de precedência. Se eu colocar uma tag b dentro de um p nada de estranho ocorrerá. Um trecho de código ficará em negrito como despertado

E se eu adicionar uma nova tag p? Ora, faz sentido para você um parágrafo dentro de um parágrafo? Pois é, também não faz sentido para o navegador, e este se encarregará de fechar o parágrafo anterior antes de abrir um novo.

Tá, mas por que tudo isso?

Lembra que eu falei acima sobre essa nova onda de geradores de blogs estáticos? O mais famoso dele - e talvez o mais antigo, até onde eu saiba - é o Jerkyll. Vejamos como é um post no Jerkyll: 

---
layout: post
title:  "Welcome to Jekyll!"
---

# Welcome

**Hello world**, this is my first Jekyll blog post.

I hope you like it!

Jerkyll e tantos outros usam Markdown como formato para se escrever conteúdo. Este formato então será preprocessado e um html será gerado. Basicamente o que acontece ali são declarações que serão usadas para a geração da página, que template usar e o conteúdo em si.

Ora bolas, eu sou programador. HTML não é sequer algo completamente difícil de entender para um leigo, imagine para mim. Vejamos então como fica o mesmo conteúdo convertido para HTML: 

<!DOCTYPE html>
<html lang="en">
<meta charset="utf-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<title>Welcome to Jekyll!</title>

<h1>Welcome</h1>
<p><b>Hello world</b>, this is my first Jekyll blog post.
<p>I hope you like it!

Convenhamos, não é lá grande coisa escrever tags HTML uma vez que temos as facilidades que apresentei acima, como deixar tags abertas e o mínimo necessário. Quanto mais conteúdo houver, menos aquele pequeno início importa. São poucas linhas que são praticamente obrigatórias em qualquer site, então elas nunca mudarão.

Um post mínimo nunca vai ser muito menos do que o código apresentado acima.

Só conteúdo não basta

Apesar de ser incontestavelmente o principal, nem só de conteúdo se faz um site. Estilo, alguma dinamicidade, elementos de navegação se fazem necessário.

Para a primeira questão, nada muito além do trivial: vamos criar uma referência para um arquivo de estilo externo que se repetirá em todas as páginas do site. Concorda comigo que esse arquivo de estilo também se repetirá invariavelmente para todas as páginas? Qualquer regra específica de formatação, usemos class ou ids.

Mas e menu? E o topo, minha logo, mensagem de copyright? Bem, para isso existe javascript. Perceba que focamos até então no essencial, que é o conteúdo. Robôs de busca, indexadores, crawlers tem uma trabalheira para limpar o conteúdo desnecessário e entender o site de forma a mostrar só o que é importante. Estamos fazendo isso "de graça" para essa legião de robôs.

Com javascript adicionemos esses elementos cosméticos. A vantagem de fazer isso dinamicamente com javascript ao invés de gerar todo o HTML é que o conteúdo principal estará completamente carregado antes mesmo de menu, logo etc. Outra vantagem é que se quisermos alterar o tema ou mudar algum elemento na tela não precisamos gerar novamente todas as páginas com essa alteração, mas apenas mudar o que o javascript vai adicionar.

Para deixar tudo preparado, adicionemos as tags link e script à nossa página: 

<!DOCTYPE html>
<html lang="en">
<meta charset="utf-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<link rel="stylesheet" href="style.css">
<script src="script.js"></script>
<title>Welcome to Jekyll!</title>

<h1>Welcome</h1>
<p><b>Hello world</b>, this is my first Jekyll blog post.
<p>I hope you like it!

Pronto, agora chegamos no nosso cabeçalho definitivo. Tudo o que está acima desta linha branca separando o conteúdo do header precisa ser repetido em todas as páginas do site. Entenda isso como uma convenção. São 7 linhas que jamais mudarão (exceto o conteúdo de title que muda para cada post). Aconselho criar um template vazio para daí copiar e criar novos posts.

Mas onde eu quero chegar?
